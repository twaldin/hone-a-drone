## 1. CLI shape

```bash
hone run --dir ./orchestrator/ \
    --grader ./graders/spawn_latency.sh \
    --grader ./graders/message_count.sh \
    --grader ./graders/kill_reliability.sh \
    --grader ./graders/task_completion.sh \
    --grader-combine pareto \
    --mutator-pool harness:claude-code:sonnet,harness:codex:gpt-5.4 \
    --budget 200
```

`--grader` becomes **repeatable** (accepts N occurrences). Each grader is
treated as an independent scoring axis.

`--grader-combine {pareto|weighted|lexicographic|epsilon}` — **default
`pareto`** per scout. Rationale: weighted-sum forces users to hand-
calibrate unit scaling across non-commensurable axes; that's a bad
default. Pareto exposes the tradeoff, lets the user pick from the
frontier post-hoc.

Combine-specific flags:
- `--grader-weights w1,w2,...` (for `weighted`)
- `--grader-order g1,g2,...` (for `lexicographic`)
- `--grader-epsilon g1=v1,g2=v2,...` (for `epsilon`)

## 2. Bandit reward for multi-grader

Single-scalar combines are easy — the scalar IS the reward.

**Pareto** is the interesting case. Reward for an iteration = candidate's
**hypervolume contribution** to the current Pareto frontier (how much new
objective-space volume it claims, in positive direction per axis). When
a candidate is dominated → reward = 0. When it expands the frontier →
reward = volume added, normalized to per-axis ranges.

This keeps the bandit's one-reward-per-iter contract intact while
still rewarding arms that push the frontier in ANY direction, not just
one.

## 4. Implementation deltas from v3.0

- `optimizer.py`: collect a vector of scores (one per grader) instead of
  a scalar each iter; dispatch combine-fn to reduce to bandit reward.
- `pareto.py` (new): maintain frontier, compute hypervolume delta,
  dominated/dominant checks.
- `leaderboard.py`: multi-axis sort + per-grader arm winners.
- `cli.py`: repeatable `--grader`; combine-specific flag parsing.

## 5. Open questions (to resolve before full v3.1 spec)

- **Hypervolume reference point.** Pareto hypervolume requires a
  reference point; choosing one per run affects reward magnitude.
  Options: fixed anchor from a baseline candidate, or dynamic anchor
  updated when frontier retreats.
- **> 3 graders.** Pareto-front grows combinatorially; scout notes
  "curse of dimensionality" kicks in above ~3 objectives. Cap or
  suggest epsilon-constraint over Pareto for 4+ graders?
- **Partial grader failures.** If one grader crashes (e.g., timeout),
  does the whole iter count as an error, or do we treat missing axes
  as worst-case? v3.0's "error = 0 reward" rule doesn't translate
  cleanly.
- **Lineage under Pareto.** GEPA keeps the single best parent. Under
  Pareto, the "parent" is a FRONTIER. Does the next iter's parent get
  sampled uniformly from the frontier, weighted by hypervolume
  contribution, or another rule?

## 5a. Secondary v3.1 feature: adaptive file weights in scheduler (opt-in)

Run-2 (2026-04-20, v2-rerun 3-way) produced a load-bearing negative
signal for the current `diagnose` scheduler's behavior on a single-file-
dominated task:

- `world_model.py`: 14 edits across runs B+C, **zero** accepted
  improvements.
- `attitude_ctrl.py`: 3 edits in run B, zero accepts.
- `planner.py`: 100% of all per-run gains attributed here (runs A/B/C
  final gains +0.183 / +0.085 / +0.128).

The scheduler is functioning as designed — `fail_class=out_of_bounds`
routes to `planner.py`, and out_of_bounds is the dominant failure — but
the non-planner edits it does schedule (via `gates_passed=0` rule, high-
latency rule) systematically waste budget on files that have no
headroom.

### 5a.1 Why this is v3.1, not v3.0

Single-task evidence. Drone-racing is planner-dominated. SWE-bench Live,
Stockfish, or an agent orchestrator may have different dominant-file
distributions or no dominant file at all. Locking in adaptive weights
now would overfit to one task's topology. Revisit after 2+ reference
problems have run.

### 5a.2 Proposed design (opt-in)

New flag: `--scheduler-adaptive-file-weights {off|accept_rate|ucb1}`,
default `off` (baseline behavior unchanged for reproducibility).

- `accept_rate`: each file's selection probability in the fallback
  target distribution scales with its cumulative `(accepts + 1) /
  (edits + pool_size)` smoothed estimate.
- `ucb1`: treat file selection itself as a bandit problem (UCB1 over
  target files) when the `diagnose` scheduler's rules don't fire.
  Reuses v3.0's selectors module — cheap to add.

Per-file stats stored in the same `bandit_state` JSON block that v3.0
already writes per-iter.

### 5a.3 Explicit carve-out for user-forced routing

Users who know their task's topology can always edit `scheduler.json`
rules to force weights. Adaptive weighting is for runs where users
don't have prior knowledge of which files will matter — it is NOT a
replacement for well-specified scheduler rules.
