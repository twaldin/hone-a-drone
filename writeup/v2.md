## Feature attribution (Question 2)

**Can't cleanly attribute the v2 gain vs v1 delta to any one feature.
The bigger issue: the grader didn't exercise v2's new features.**

My `run_rollout.py` loads ONLY `controllers/planner.py` via `importlib`.
`attitude_ctrl.py`, `baseline.py`, `state_estimator.py`, `gate_detector.py`,
`world_model.py` are never imported by the graded code path. This is a
v1-era constraint (CLAUDE.md explicitly required `planner.py` to be
self-contained) that I inherited into v2 without rearchitecting.

Direct consequence for v2:

| Feature | Exercised on this benchmark? |
|---      |---                           |
| `--dir` multi-file target | No — 5 of 6 targetable files are grader-inert |
| Dynamic (diagnose) scheduler | Partially — it concentrated 96/98 iters on `planner.py`, which is correct policy. But since only one file matters, the scheduler was picking the only useful target, not actually routing. |
| ACE observer | Yes — fires produced real rules, some of which the mutator appeared to absorb (iter 53 win came ~3 iters after observer fire #5). But rules also appeared to over-constrain late exploration (iters 54-100 = 0 wins). |

So the honest answer: **this benchmark is single-file. A single-file-plus-
scheduler config is literally a single-file config. v2's dir-mode machinery
had nothing to lift.** v2's *thesis* — that multi-file + ACE + scheduler
beats single-file — is **untested** by this run, not refuted.

### Why v2 still lost to v1 under-the-hood

Three real factors beyond "grader unfair":

1. **24% edit-mode crash rate bled $$$.** `claude-code -p
   --dangerously-skip-permissions` with the Edit tool in a 6-file workdir
   exited with code -1 (likely timeout at harness's 300s default, possibly
   native crash) on 24 of 100 iterations. v1's text-response mode had 0%
   errors. Net effective iterations: v2 ~76 vs v1 13, but the per-effective-
   iter value is lower in v2 because of additional workdir-clutter overhead
   (below). At $0.36/iter average, 24 errors = ~$8.64 direct burn.

2. **Workdir clutter.** v2 mutator gets a populated workdir + Read/Glob/Bash
   tools + a CLAUDE.md with ACE-accumulated rules. v1 mutator gets one file's
   text in the prompt. More degrees of freedom → more inference tokens spent
   exploring → higher timeout risk → more regressions per accepted mutation.
   This is a discipline problem, not a mode problem. Edit-mode multi-tool
   is the *right* long-term direction (it's how AI coding CLIs are designed
   to work); it's just not paying off on a benchmark where the extra scope
   is illusory.

3. **ACE rules may have over-constrained late exploration.** Five observer
   fires → 13 rules accumulated. Iter 53's win happened after 5 fires worth
   of rules were in place, so the rules didn't *block* the win. But iters
   54-100 produced zero further improvements despite 45 more attempts,
   consistent with "exploration space shrank faster than rules bought useful
   constraint." No hard proof — this is speculation from score flatness.
   Could also just be hitting a local optimum the mutator can't escape
   regardless of rules.

## Updated ACE verdict (Question 3)

**v1 verdict was: ACE not justified yet (70/30 LLM-catchable/human-needed split
on 13 iters, 0 repeat-pattern observations). Threshold for "justified" was
80/20 or clear repeat-pattern evidence.**

**v2 doesn't cleanly move that verdict** — because v2 couldn't fairly test it.
But it produces one new data point: **the ACE observer's produced rules are
substantive**, not noise. It's doing real reflection. Whether the rules
actually *lift scores* on a problem that genuinely needs multi-file coherence
is still an open question.

New split, from v2 observer fire data specifically:

- 5 observer calls, all applied=True (zero parse failures, zero rollbacks)
- 13 rules produced; by inspection, ~9 are substantive engineering insights
  ("don't flip cubic spline BC type"), ~4 are defensive restatements of
  existing constraints
- ACE infrastructure is robust; rule quality is high; the question is whether
  the rules actually change mutator behavior in a score-moving way

Hypothesis for v3: if ACE is tested on **exactly v1's configuration** (single-
file, text-mode), and it beats v1's 1.0778, then ACE pulls its weight
independent of multi-file. If it doesn't, the v2 verdict compounds with v1's:
ACE is not yet earning its keep on this class of problem.

## V3 plan

Three experiments, in order:

### V3 Exp A — ACE alone, on v1's clean setup
The cleanest test of whether ACE earns its keep.

- Single-file mode: `hone run controllers/planner.py`
- v1's text-response mutator path (zero edit-tool flakiness)
- Add observer support to `optimize()` (currently only in `optimize_dir()`)
- Observer fires every 10 iters; same Reflector/Curator pipeline
- Same seed, budget 100

Target: **beat v1's 1.0778**. If yes, ACE is validated on focused problems.
If no, ACE is not pulling its weight and needs architectural rethink.

### V3 Exp B — fix the 24% edit-tool crash rate
Parallel investigation. Can run while Exp A runs.

- Grep all 24 error iters for timing patterns, prompt length, target line
- Probe harness edit-mode in isolation with a synthetic planner.py prompt
- Check if exit code -1 is SIGKILL at 300s (timeout) or native crash
- Fix options: bump timeout, add retry-on-exit-1, upstream to claude-code

### V3 Exp C — make grader actually multi-file
Only if Exp A + Exp B don't close the gap.

- Restructure `run_rollout.py` to load `baseline.py` as the entry point
- `baseline.py` already imports `controllers.planner.Planner`; extend to
  actually *use* helpers from `attitude_ctrl.py`, `state_estimator.py` etc.
- Relax the CLAUDE.md "planner.py self-contained" rule
- Re-run v2 config against the new grader. This is the fair test of v2's thesis.

## TL;DR

v2 ran to completion. Scored +11.7% over seed, 16 points behind v1. The
v2-specific features (multi-file, scheduler, ACE) were not fairly tested
because the grader loads only `planner.py`. ACE observer produced substantive
rules and its infrastructure is robust. Edit-tool had a 24% crash rate that
bled budget. V3's first test is the cleanest: ACE alone, on v1's setup, see
if it lifts v1's 1.0778.
